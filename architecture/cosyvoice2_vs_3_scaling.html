CosyVoice 2 vs CosyVoice 3

This page answers one narrow question: does `CosyVoice 3` actually solve the `speech token -> mel` scalability problem, or is it a better implementation of the same family? The answer from the current paper, repo metrics, and local code is: better stack, not full structural fix.

What clearly improved

Better published quality metrics, stronger streaming / deployment story, a simpler token-side expansion path, and a `DiT`-based flow estimator instead of the older causal U-Net-like estimator.

What still remains

An autoregressive speech-token LM, iterative flow decoding with `n_timesteps=10`, `mel` output before vocoding, and therefore the same double-serial structure we are worried about.

Scalability Verdict

Better

Deployment

The project claims bi-streaming latency as low as `150 ms`, which suggests the runtime pipeline improved meaningfully.

Better

Estimator Design

The older flow estimator is replaced by a `DiT`-based estimator, which is a substantial architectural cleanup inside the same family.

Still There

AR Token LM

The upstream speech-token model is still autoregressive, so one serial stage remains before acoustic decoding even begins.

Still There

Iterative Flow

The flow path still runs iterative decoding and still produces `mel`, then sends that into a separate vocoder.

Architecture Difference

The most important distinction is not whether `CosyVoice 3` is better. It is. The question is whether it removes the structural bottleneck. It does not.
CosyVoice 2

Older Flow Stack

Stage 1 Autoregressive Speech-Token LM

Still token-by-token generation upstream.

Stage 2 Upsample Conformer Encoder

Token-rate hidden states are expanded to mel-rate `mu`.

Stage 3 Causal U-Net-like Flow Estimator

`ResNet1D + Transformer` estimator inside iterative CFM decoding.

Stage 4 `10`-step Flow -> Mel -> HiFT

Iterative mel generation, then separate vocoding.

CosyVoice 3

Newer Flow Stack

Stage 1 Autoregressive Speech-Token LM

Still present. Better LM family, but still serial.

Stage 2 Pre-Lookahead + Repeat-Interleave

Simpler token-side expansion instead of the older upsample Conformer path.

Stage 3 `DiT` Flow Estimator

Cleaner transformer-style estimator for the same iterative flow role.

Stage 4 `10`-step Flow -> Mel -> CausalHiFT

Still iterative mel decoding followed by a separate vocoder.

Main Takeaways

Takeaway

Quality is clearly up

The published table shows meaningful gains over `CosyVoice2` on Chinese CER, English WER, and speaker similarity.

Takeaway

Implementation is cleaner

`CosyVoice 3` looks like a smarter token-to-mel implementation, especially around the expansion path and estimator design.

Takeaway

No isolated speed proof

The public material does not give a clean “the flow estimator is X% faster” result, so we should not pretend it does.

Takeaway

Not the final scalability answer

The double-serial structure still exists: AR speech-token generation followed by iterative flow decoding.
