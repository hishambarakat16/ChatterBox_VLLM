Original Failure
old baseline and streaming paths broke at concurrency >= 2
Baseline hit tensor-size mismatch. Streaming could avoid a Python exception but still collapsed one request to a tiny output.
First Correctness Win
concurrent restored correctness through concurrency=4
The fix was request-local T3 backend/analyzer state plus a coarse full-decode lock.
First Real Scaling Win
scheduled replaced the coarse lock with batched T3 cohorts
First measured batch proof: run_cohort requests 2 and prefill.batch (4, 72, 1024).
Latest Planner Win
Hydra h2 now leads Medusa in single-request speculative planner speed
Current best measured setting is Hydra k3 with 24.34% planner speedup, acceptance=0.6078, exact_token_match=true, and rebuild_count=0. Next step is scheduled runtime integration and concurrency validation.
New Turbo S3 Result
Scheduled + Hydra + turbo S3 dramatically shrinks `token2mel` cost
The new runtime path is stable through `c8`; `stage_s3_token2mel_s` now stays below 1.8s while `stage_s3_s_mean` is 0.8822s at c2.

Concurrent Runtime

Make T3 Safe With Request-Local Decode State

correctness gain
coarse lock
What Changed
  • Built a request-local T3 backend instead of reusing one shared backend object.
  • Made the alignment analyzer request-local and removable on close.
  • Wrapped the full T3 decode in one coarse lock so two requests could not corrupt each other.

Alignment Cleanup

GPU-Local Scheduled Alignment State

quality preserved
throughput gain
not final
What Changed
  • Kept the scheduled alignment guard enabled after the sweep showed it is necessary for output quality.
  • Removed the scheduled hook's per-step .cpu() attention copy.
  • Removed the growing CPU-side full alignment matrix.
  • Replaced it with rolling GPU-local state: recent rows, early-text activation max, post-completion tail mass, and post-completion repetition mass.

Turbo S3 Addition
Introduced the `scheduled_turbo_s3` runtime path and benchmark harness; c1–c8 throughput shifted because `token2mel` now runs in the new turbo mode with per-stage metrics published in the trace run results.
Latest interpretation: request-local runtime state is no longer the main unresolved issue, coarse full-decode locking is gone, real batched T3 scheduling exists, and the newest alignment cleanup removed a real chunk of guard overhead. Hydra now leads the planner benchmark, so the next technical question is scheduled runtime integration and concurrency validation for Hydra, plus continued deeper T3 profiling on the remaining attention-output fallback cost.
Current Plain-English Read
  • The first fixes were about state ownership, not speed.
  • The first big correctness win came from making T3 decode state request-local.
  • The first big scaling win came from replacing the coarse lock with batched T3 cohorts.
  • The latest guard cleanup kept alignment on, moved rolling state back onto GPU, and cut a meaningful chunk of scheduled guard overhead.
  • The latest planner improvement is Hydra, which beats Medusa on single-request speculative planner speed.
Still Open
  • integrate Hydra into scheduled runtime and measure baseline vs Hydra under the same concurrency harness
  • validate staggered-arrival behavior under live load, not just simultaneous starts
  • profile deeper inside T3 prefill and decode compute
  • measure how much of the remaining guard cost comes from the output_attentions=True fallback path
  • revisit S3 only after the deeper T3 read is clear
Best One-Line Summary
The runtime evolved from “shared mutable inference state breaks concurrency” to “batched T3 scheduling scales meaningfully through concurrency=8; now Hydra is the leading speculative planner component and the next proof target is scheduled-runtime integration with real concurrency gains.”
