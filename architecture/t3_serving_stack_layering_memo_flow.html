T3 Serving Stack Layering

Visual companion to the memo: four-layer stack, valid serving combinations, current path, and the four future paths (`keep custom scheduler`, `vLLM`, `SGLang`, `TensorRT-LLM`). Use this board to see where the new Hydra + turbo S3 stack sits relative to those alternatives.
Core message
Don’t layer all serving engines; pick one
Hydra + turbo S3 keep the renderer fixed while you can swap the scheduler/runtime engine as a single block.
Current weakness
T3 scheduler
The memo highlights the custom scheduler as the main pain point, not the downstream turbo S3 renderer.
Candidate replacements
vLLM / SGLang / TensorRT-LLM
Each claims the scheduling layer; only TensorRT-LLM also overlaps with the execution backend.
Layer Model
Four layers, their responsibilities, and who currently owns each.
Layer Responsibility Current owner Candidate replacements
App / Adapter Session setup, request reception, `T3->S3` handoff Custom Chatterbox runtime Keep custom
Scheduling / Serving Policy Batching, admission, fairness, prefill/decode policy Custom T3 scheduler vLLM / SGLang / TensorRT-LLM
Execution Backend Kernels, KV cache, CUDA overlap PyTorch + custom runtime TensorRT-LLM, parts of vLLM/SGLang
Renderer Speech tokens -> mel -> waveform Turbo S3 Keep current for now
  • TensorRT-LLM straddles both scheduling and execution, unlike vLLM/SGLang (primarily full serving engines).
  • Hydra + turbo S3 keep the renderer untouched while experimentation focuses on the scheduling layer.

    How a request flows today through the custom stack.

    Client / Adapter

    Handles sessions, options, Hydra checkpoint selection, pyro logging.

    Custom T3 Scheduler

    Batching, hydra speculative rounds, alignment guard, request locking.

    T3 + Hydra Decode

    Generates CFG-duplicated tokens, Hydra future heads, speculative acceptance.

    Turbo S3 Renderer

    Token-to-mel flow + HiFT vocoder still untouched.

    • Future Options
    Four clear paths for evolving the scheduler/runtime.
    Option What changes Pros Cons
    Option A — Improve custom scheduler Keep adapter + scheduler, tighten policy, better cohort merging Lowest risk, Hydra/CFG semantics unchanged More scheduler engineering, limited until runtime matures
    Option B — Swap in vLLM Thin adapter feeds vLLM; vLLM handles scheduling, Hydra decode ported Mature serving engine, batching and KV management built-in Nonstandard T3 path; Hydra/CFG work may require extra engine effort
    Option C — Swap in SGLang Adapter + SGLang runtime handle scheduling + decode Modern serving features, router/PD ecosystem Less conservative than vLLM; requires porting work
    Option D — Swap in TensorRT-LLM Adapter calls TensorRT-LLM; backend provides NV-optimized kernels + scheduling Strongest NVIDIA performance, IFB, paged attention, overlap Highest porting cost for Hydra + CFG; best later
    Combination Rules
    Which stacks are valid, and which mixes should be avoided.
  • Choose one serving engine per request path; don’t layer vLLM + SGLang or vLLM + TensorRT-LLM together.
  • Custom adapter + vLLM/SGLang/TensorRT-LLM is a valid pattern if you only replace the scheduling layer.
  • Keep turbo S3 downstream unless you plan an explicit renderer migration as well.
