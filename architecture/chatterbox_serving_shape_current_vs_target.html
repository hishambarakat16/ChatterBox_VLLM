Sequential vs Concurrent Start-Fresh Calls
Reasonable for sequential single-request inference
Request A
Build self.conds, build analyzer/backend, decode, return, then only after that Request B starts.
Request B
Can safely "start fresh" because nothing else is mutating the shared object at the same time.
Unsafe when two requests overlap on one shared model object
Request A
Writes self.conds, installs analyzer A, stores backend A on self.patched_model.
Request B
Starts before A finishes, rewrites self.conds, installs analyzer B, stores backend B on the same self.patched_model.
Request A
Continues decode while reading shared object state that may now belong partly to Request B.

3. Current Bottleneck Interpretation
1

T3 is the first correctness blocker for concurrency.

Before throughput matters, shared mutable T3 inference state makes concurrency=2 unsafe. The first engineering question is still correctness on one shared worker.
2

S3 is still the first likely performance hot path after correctness is fixed.

S3 remains batch-size-1 oriented, runs on the longer mel timeline, and contains the token-to-mel plus vocoder cost. Once T3 stops corrupting concurrent runs, profile here first.
3

forcing EOS is not automatically an error.

Healthy single-request baseline and streaming traces both hit forcing EOS and still completed with plausible output. Treat it as a warning signal only when it coincides with collapse or shape/runtime failures.

5. Per-File Rewrite Map
File Current Responsibility Problem Intended Change Expected Boundary After Redesign
mtl_tts.py Top-level orchestration, prompt conditioning, shared self.conds, direct T3/S3 calls, watermark. Shared model object currently owns request-local conditionals and exaggeration mutations. Become a thin API/front-door wrapper that creates request context and delegates to runtime worker. Inputs: raw request fields. Outputs: request context or final waveform. No persistent request state on the shared object.
runtime/worker.py Current Layer 1 wrapper that clones conditionals and exposes session-based generation. Helps with prompt-state isolation but does not yet own full scheduler logic or fix shared T3 internals. Own request/session creation, scheduler state, active request tracking, and safe invocation of shared workers. Inputs: request context + text/options. Outputs: safe shared execution and final request result.
t3.py Shared weights plus backend creation, analyzer creation, and decode loop. Request-local mutable inference state is stored on shared self. Keep weights on self; move backend, analyzer, caches, and decode progress into request-local inference context. Inputs: t3_cond, text_tokens, request-local decode state. Outputs: speech tokens plus updated request-local state.
alignment_stream_analyzer.py Request-level alignment heuristics implemented through hooks on shared transformer layers. Hooks and config mutation are global side effects on shared layers. Make analyzer fully request-local and detachable, or remove shared hooks in concurrent mode until a safe replacement exists. Inputs: request-local attention/logit stream. Outputs: request-local alignment state and optionally sanitized logits. No lingering global hooks.
s3gen.py Speech-token filtering, token-to-mel flow decode, HiFT waveform synthesis. Still batch-size-1 oriented and still the first likely performance hot path once correctness is fixed. Keep interface stable first; treat as shared worker with request-local inputs, then profile and optimize after T3 correctness. Inputs stay speech_tokens + ref_dict. Outputs stay output_mels and output_wavs.
6. What We Need To Preserve

Do not copy model weights per request.

The target is one shared T3/S3 worker per model instance, not one model copy per concurrent request.

Only copy request-local mutable decode state.

Clone conditionals and keep caches, token history, and analyzer state inside request contexts only.

Keep the current tensor interface valid.

Inputs and outputs should still line up with the known-good trace path: text_tokens -> speech_tokens -> output_mels -> output_wavs.

Preserve the known good single-request flow.

Sequential single-request inference already works. The redesign should keep that behavior while making concurrent requests safe.
