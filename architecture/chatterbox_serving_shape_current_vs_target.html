Serving Architecture

Current Chatterbox shape vs the streaming shape we actually need

The current problem is not one slow function. It is a shared-runtime problem plus a double-serial model pipeline. On the left is how the open-source code currently behaves. On the right is the target runtime shape: keep one GPU-resident worker with shared read-only weights, move request state into explicit session objects, then profile whether S3 is still the first true hot path.

Current failure mode

self.conds lives on the shared wrapper, so prompt-derived request state is mutated on the model object itself. That makes one shared instance awkward or unsafe for many active requests.

Current model cost

The path is serial twice: T3 autoregressively emits speech tokens, then S3 iteratively decodes on the longer mel timeline before HiFT turns mel into waveform.

Target decision

Fix serving shape first. If concurrency is still poor after request/session isolation, then redesign S3. Do not confuse runtime cleanup with solving the deeper serial model problem.

Target Serving Shape

Shared worker, explicit sessions

Target runtime from CHATTERBOX_FORK_IMPLEMENTATION_PLAN.md. This is a serving architecture change first, not yet a deep model rewrite. The point is to make one GPU-resident worker safely host many sessions, then measure whether S3 still dominates.

1. Shared worker owns weights only
fork plan: 67-77

A worker object owns the shared T3, S3, tokenizer, vocoder, and reusable helpers. Request-specific state is not stored on the wrapper.

one GPU copy of weights scheduler-friendly
2. Create one session per request
fork plan: 55-65, 121-136

A session object owns prompt conditionals, request settings, decode progress, caches, and offsets. That is the boundary we are missing today.

request isolation explicit state
3. Pass conditionals explicitly
fork plan: 98-125

The streaming path should pass T3 conditioning and S3 reference state through the session, not through self.conds on a shared wrapper.

no hidden mutation clean ownership
4. Keep T3 and S3 behavior mostly the same first
fork plan: 79-92, 159-181

Do not fork deep model files yet unless runtime cleanup is not enough. Preserve baseline behavior so the next measurement tells us whether the problem was runtime shape or true model cost.

baseline preserved model surgery later
5. Measure concurrency on the clean runtime
checklist: 15-35

After request isolation, profile first-chunk latency, inter-chunk latency, full latency, VRAM, and max concurrent sessions per GPU. Then decide whether S3 must change.

high-signal baseline GPU density focus
6. Attack S3 only if still justified
checklist: 31-41

If concurrency is still poor on the cleaned runtime, then redesign or replace S3. Only revisit T3 after that.

runtime first S3 next

The main GPU gain is safe weight sharing across many sessions, not magic speedup from one function.

  one request -> one model instance

Important limit

What this does not fix yet

Even after the runtime cleanup, the model family is still serial twice.

  text -> T3 autoregressive tokens

  mel -> vocoder

So runtime cleanup is necessary,
but it is not the final scaling solution.

Main takeaways

Why current shape fails

  • Request state is hidden on the shared wrapper via self.conds.
  • That invites cross-request mutation or forces wasteful one-model-per-request serving.
  • The actual inference path is also serial twice, so concurrency pressure compounds quickly.

Why target shape is better for GPUs

  • One worker can keep one copy of model weights resident on GPU.
  • Each active request carries only its own lightweight session state and caches.
  • That gives a clean platform for scheduling, batching experiments, and honest VRAM accounting.

What the next bottleneck test means

  • If concurrency improves enough, the runtime shape was a major part of the issue.
  • If it still does not scale, then S3 is justified as the first model-level redesign.
  • Only after that should T3 be reconsidered.
