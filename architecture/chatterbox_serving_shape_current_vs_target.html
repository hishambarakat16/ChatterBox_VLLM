Sequential vs Concurrent Start-Fresh Calls
Reasonable for sequential single-request inference
Request A
Build self.conds, build analyzer/backend, decode, return, then only after that Request B starts.
Request B
Can safely "start fresh" because nothing else is mutating the shared object at the same time.
Unsafe when two requests overlap on one shared model object
Request A
Writes self.conds, installs analyzer A, stores backend A on self.patched_model.
Request B
Starts before A finishes, rewrites self.conds, installs analyzer B, stores backend B on the same self.patched_model.
Request A
Continues decode while reading shared object state that may now belong partly to Request B.

3. Current Bottleneck Interpretation
1

Baseline and streaming still fail the concurrency-safety test.

The original shared-state bug is still real in the old paths. That is no longer the main problem in the validated runtime branch.
2

The scheduled path is the first real batched T3 serving step.

It keeps one shared T3 weight copy, keeps per-request mutable decode state separate, and batches multiple requests together for T3 prefill and decode. The hardest evidence is still strongest for same-time arrivals.
3

The next problems are staggered-arrival validation, then S3 throughput and broader telemetry.

The shared-state corruption bug is no longer the main blocker. Current read:
c2 is the sweet spot, c4 stays correct but adds almost no throughput, and S3 is now a more plausible next hot path.

forcing EOS is still not automatically an error by itself.
4. Current Scheduled Runtime And Target Boundary
Shared weights stay shared. Mutable decode state moves into request-local contexts. Scheduler state sits outside the model objects. The currently validated runtime is the scheduled path: compatible requests are grouped into one T3 cohort, prefilled together, then stepped together while each request keeps separate decode state and output.

5. Per-File Rewrite Map
File Current Responsibility Problem Intended Change Expected Boundary After Redesign
mtl_tts.py Top-level orchestration, prompt conditioning, shared self.conds, direct T3/S3 calls, watermark. Shared model object currently owns request-local conditionals and exaggeration mutations. Become a thin API/front-door wrapper that creates request context and delegates to runtime worker. Inputs: raw request fields. Outputs: request context or final waveform. No persistent request state on the shared object.
runtime/worker.py
runtime/worker_scheduled.py		 Build request-local conditionals, create sessions, and hand requests to the scheduled shared-worker path. The runtime boundary is now split across base worker helpers and the scheduled worker, so ownership has to stay explicit. Keep request/session creation in the worker layer and keep scheduler-owned batching out of model modules. Inputs: request fields plus shared weights. Outputs: isolated request context, queued decode work, and final request result.
t3.py Core shared T3 weights and embedding interface for the multilingual speech-token LM. The old path stored request-local inference state on shared self; the remaining gap is a cleaner native prefill/step boundary. Keep weights on self; keep backend state, analyzer state, caches, and decode progress in request-local scheduled decode objects. Inputs: t3_cond, text_tokens, request-local decode state. Outputs: speech tokens plus updated request-local state.
alignment_stream_analyzer.py Baseline alignment heuristics implemented through hooks on shared transformer layers. In the old path, hooks and config mutation are global side effects on shared layers. Keep analyzer behavior request-local inside scheduled decode state and remove reliance on shared lingering hooks. Inputs: request-local attention/logit stream. Outputs: request-local alignment state and optionally sanitized logits. No lingering global hooks.
s3gen.py Speech-token filtering, token-to-mel flow decode, HiFT waveform synthesis. Still batch-size-1 oriented and now the clearest likely performance hot path after the scheduled T3 work. Keep interface stable first; treat as shared worker with request-local inputs, then profile and optimize it after the T3 scheduler work. Inputs stay speech_tokens + ref_dict. Outputs stay output_mels and output_wavs.
6. What We Need To Preserve

Do not copy model weights per request.

The target is one shared T3/S3 worker per model instance, not one model copy per concurrent request.

Only copy request-local mutable decode state.

Clone conditionals and keep caches, token history, and analyzer state inside request contexts only.

Keep the current tensor interface valid.

Inputs and outputs should still line up with the known-good trace path: text_tokens -> speech_tokens -> output_mels -> output_wavs.

Preserve the known good single-request flow.

Sequential single-request inference already works. The redesign should keep that behavior while making concurrent requests safe.
